# Man-in-the-middle in the juliaup AUR deployment workflow could allow corruption of deployed binaries

The GitHub Actions workflow that juliaup uses to deploy to the Arch User Repository (AUR) disabled SSH host key checking (`StrictHostKeyChecking=no`) when connecting to `aur.archlinux.org`. An attacker capable of performing a man-in-the-middle attack on the deploy job's connection to the AUR could serve malicious content that is executed during package building (the PKGBUILD sources are evaluated as part of the build), thereby gaining arbitrary code execution in the CI job context.

### Impact

The deploy job has access to the AUR deploy secret. An attacker who achieved code execution in this context could corrupt the binaries published to the AUR, potentially affecting users who install Julia via juliaup through the Arch User Repository. Mounting the attack requires the ability to MITM the network path between the GitHub-hosted CI runner and the AUR, which is a significant barrier, and there is no indication of active compromise.

### Patches

The issue is fixed in [JuliaLang/juliaup#1521](https://github.com/JuliaLang/juliaup/pull/1521), which:

  - Re-enables SSH host key verification by vendoring the verified `aur.archlinux.org` host keys (ED25519, ECDSA, and RSA) in `.github/aur_known_hosts`.
  - Replaces the workflow's blanket `permissions: write-all` with least-privilege scopes, restricting `contents: write` to only the release-publishing jobs for additional hardening.

### Remediation

The issue has been resolved. Because this is an infrastructure issue, there is no user-side remediation.
